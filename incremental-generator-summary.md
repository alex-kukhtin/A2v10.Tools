# Інкрементальний генератор: підсумок дизайну

Документ підсумовує рішення, ухвалені під час обговорення `IIncrementalGenerator`
для проєкту з двома потоками генерації: **компаньйони для класів** та
**ембедер ассетів**. Акцент — на потоці обробки C#-файлів і зв'язуванні класів
з `metadata.json`.

---

## 1. Загальна структура

В одній бібліотеці-аналізаторі (`netstandard2.0`) живуть **два окремі генератори**
(два класи з `[Generator]`):

- **Генератор A — компаньйони.** Знаходить позначені атрибутом `partial`-класи,
  для кожного бере сусідній `metadata.json`, генерує файл-компаньйон, а також
  спільний клас-реєстр.
- **Генератор B — ембедер ассетів.** Бере файли ассетів, стискає їх і записує
  у вигляді C#-коду. Уже готовий — у цьому документі лише згадується.

Потоки не ділять ані вхід, ані модель, ані вихід — тому це **два класи**, а не
один генератор із двома гілками. Спільне в них лише те, що обидва —
`IIncrementalGenerator`.

---

## 2. Обробка C#-файлів (генератор A)

### 2.1. Пошук класів — через атрибут, не через `partial`

Відбір класів робиться **не** за наявністю модифікатора `partial` (це слабкий
сигнал — `partial` використовують і поза контекстом entity), а за **маркерним
атрибутом** через `ForAttributeWithMetadataName`.

Це найдешевший із синтаксичних провайдерів: Roslyn уміє повністю пропускати
файли, де потрібного атрибута немає. Правка звичайного `.cs`-файлу без атрибута
майже не зачіпає генератор.

У цільовому проєкті це виглядає так:

    [ServerLogic]
    partial class Agent { }

`ForAttributeWithMetadataName` — це метод **на `SyntaxValueProvider`**, не на
самому контексті. Шлях доступу: `context.SyntaxProvider.ForAttributeWithMetadataName(...)`.
Три аргументи*:*

- **повне metadata-ім'я атрибута** рядком (з суфіксом `Attribute`, з namespace) —
  це той самий рядок-«джерело істини» з розділу 5;
- **предикат** — швидка синтаксична перевірка вузла (зазвичай «це
  `ClassDeclarationSyntax`»), щоб відсіяти атрибут не на класі;
- **трансформація** — отримує `GeneratorAttributeSyntaxContext`, і саме тут із
  символу витягуються рядки (див. 2.2).

Повертає `IncrementalValuesProvider<T>`, де `T` — легка value-модель. Доступний
з Roslyn 4.3 (VS 2022 17.3 / .NET SDK 6.0.4xx+); на старіших пакетах
`Microsoft.CodeAnalysis.CSharp` методу немає.

### 2.2. Що дістаємо з класу

Для генерації компаньйона реально потрібні лише:

- **ім'я класу** (коротке) та **namespace** — компаньйон є другим `partial`-куском
  і має потрапити в той самий namespace;
- **директорія файлу** — для зв'язування із сусіднім `metadata.json`.

Важливо: у трансформації `ForAttributeWithMetadataName` з символу типу
**витягуються рядки** (ім'я, namespace, шлях), і далі по pipeline їде модель із
рядками — **не** `INamedTypeSymbol`, **не** `ClassDeclarationSyntax`. Символ
важкий, тримає посилання на компіляцію, у pipeline йому не місце.

У трансформації два головні входи `GeneratorAttributeSyntaxContext`:

- `TargetSymbol` — символ позначеного типу (семантика): з нього ім'я та
  namespace. Каст до `INamedTypeSymbol` безпечний, бо предикат уже відсік усе,
  крім оголошень класів.
- `TargetNode` — синтаксичний вузол: з нього шлях до файлу через
  `SyntaxTree.FilePath`.

Модель класу — звичайний `record` з трьох рядків (ім'я, namespace, директорія).
Це «добрий» випадок розділу 4.3: усі поля скалярні, авторівність `record`
коректна, нічого руками писати не треба.

### 2.3. Шлях до файлу

Шлях до `.cs`-файлу доступний через `SyntaxTree.FilePath` синтаксичного вузла.
Від шляху файлу до сусіднього `metadata.json` — звичайний `Path.GetDirectoryName`
плюс ім'я файлу.

## 3. Блок `Combine`: зв'язування класів і `metadata.json`

### 3.1. Чому файл не можна читати напряму

`metadata.json` **не можна** читати з диска через `File.ReadAllText`. Не тому, що
це фізично не спрацює — спрацює, — а тому, що це ламає інкрементальну модель:
Roslyn не відстежує таке читання, кеш не інвалідовується, і результат
розходиться між IDE та CLI-збіркою. Розширені правила аналізатора (RS1035) саме
це й ловлять.

Тому `metadata.json` має заходити в генератор **легально — через
`AdditionalTextsProvider`**. Файли мають бути оголошені як `<AdditionalFiles>`
у `.csproj`.

Та сама заборона стосується **всіх** обхідних шляхів до файлової системи:
`Directory.GetCurrentDirectory`, пошук `.csproj` угору по дереву,
`Assembly.Location`. Усе це Roslyn не відстежує. Легальні канали лише два:
вміст файлів — через `AdditionalTextsProvider`; властивості проєкту — через
`AnalyzerConfigOptionsProvider` (див. 3.5).

### 3.2. Два потоки, які мають зустрітися

- **Потік A** — позначені атрибутом класи (`ForAttributeWithMetadataName`).
  З кожного: ім'я, namespace, **директорія файлу**.
- **Потік B** — усі `metadata.json` (`AdditionalTextsProvider`).
  З кожного: **директорія файлу** + розпарсена модель JSON.

### 3.3. Як зв'язувати

Потоки з'єднуються за спільним ключем — **директорією**. Технічно: один потік
матеріалізується через `.Collect()`, далі `.Combine(...)` із другим, і в
трансформації клас зіставляється зі своїм JSON за збігом папки. Концептуально —
це join двох колекцій за шляхом каталогу.

`classes.Combine(jsonFiles.Collect())` віддає пару
`(ClassModel, ImmutableArray<JsonModel>)` — кожен клас склеєний з **усією**
колекцією JSON. Вибір потрібного JSON за збігом папки відбувається вже в
наступному `Select`. Тип на виході цього `Select` залежить від правила
зіставлення (див. 3.4, відкрите питання 1): сигнатура «пара клас+JSON»
передбачає відповідь «рівно один».

На виході зв'язки — пара «модель класу + модель JSON», і вже вона йде в
`RegisterSourceOutput`. До моменту генерації жодного парсингу й жодних символів —
лише дві готові легкі value-моделі.

### 3.4. Рішення, які треба ухвалити свідомо

- **Нормалізація шляхів.** Порівнювати директорії рядковою рівністю небезпечно:
  регістр (Windows), роздільники, `..`, симлінки. Зіставлення має йти за
  канонізованим шляхом, інакше join мовчки промахуватиметься.
- **Правило зіставлення «клас ↔ json».** Однозначне лише якщо в кожній папці
  рівно один клас і рівно один `metadata.json`. Треба явно визначити поведінку
  для випадків: JSON немає; кандидатів кілька. *(Залишилось відкритим.)*
- **Інкрементальність зв'язки.** `.Combine` з `.Collect()` означає, що зміна
  будь-якого `metadata.json` перераховує всі зіставлення. Тому модель, що йде в
  `.Collect()`, має бути легкою та зі значущою рівністю.

### 3.5. Властивості проєкту через `AnalyzerConfigOptionsProvider`

Якщо генератору потрібні **властивості проєкту** (директорія, root namespace
тощо) — легальний канал це `AnalyzerConfigOptionsProvider`, не файлова система.
MSBuild-властивості пробрасуються в генератор лише якщо явно оголосити їх у
`.csproj` через `CompilerVisibleProperty`:

    <ItemGroup>
      <CompilerVisibleProperty Include="ProjectDir" />
    </ItemGroup>

У генераторі властивість читається з `GlobalOptions` з обов'язковим префіксом
`build_property.` (контракт між `CompilerVisibleProperty` і `GlobalOptions`):
`build_property.ProjectDir`. Це `IncrementalValueProvider` — одиничний, бо
властивість одна на компіляцію; далі підмішується до потоків через `.Combine`.
`TryGetValue` може повернути `false` — фолбек обов'язковий.

`CompilerVisibleProperty` — лише для **властивостей**, не для вмісту файлів.
Вміст файлів — завжди `AdditionalTextsProvider`.

## 4. `record`, обгортка та кеші Roslyn

### 4.1. Принцип кешування Roslyn

Кеш Roslyn тримається на **рівності за значенням**. Коли стадія pipeline
видає об'єкт-модель, Roslyn порівнює його з кешованою версією через `Equals`.
Якщо рівні — наступні вузли пропускаються. Якщо порівняння хибне — кеш не
спрацьовує, і генератор працює «наскрізь» щоразу.

> В активній IDE-сесії генератор може запускатися сотні разів за хвилину.
> Тому модель у pipeline має бути якомога меншою і коректно порівнюваною.

По pipeline між стадіями возимо **лише легкі value-моделі** — ані `SourceText`,
ані `AdditionalText`, ані `ISymbol`, ані `SyntaxNode`.

Лямбди в pipeline варто робити `static` — щоб гарантовано не захопити `this` чи
локальну змінну. Захоплення ламає кеш так само тихо, як важка модель.

### 4.2. Парсити JSON — рано

JSON парситься **рано**, у `Select` одразу за `AdditionalTextsProvider`, а не у
фінальному `RegisterSourceOutput`. Якщо везти сирий текст і парсити при
генерації — Roslyn кешує *текст*: будь-яка незначуща зміна (пробіл,
переформатування) інвалідовує стадію. Якщо парсити рано — кешується *модель*,
і переформатування JSON без змін по суті не перезапускає генерацію.

Результат парсингу — **власний `record`** із потрібними полями, **не**
`JsonDocument` / `JObject` / словник (вони порівнюються за посиланням, до того ж
`JsonDocument` — `IDisposable`).

### 4.3. Пастка вкладених колекцій

Структура моделі:

    {
      "id": "test",
      "fields": [ { "name": "...", "type": "..." }, ... ]
    }

`record` генерує `Equals`, який порівнює кожне поле через його власний `Equals`.
Для `String`, `Int32`, `Boolean` — це порівняння за значенням (добре). Для
**масиву** (`Field[]`) чи `List<Field>` власний `Equals` — **посилальний**: два
масиви з однаковим вмістом вважаються різними.

Наслідок: модель-`record` із полем `fields`-масивом майже завжди буде «не рівна»
сама собі між прогонами → кеш не працює.

Важливо: ламає саме **масив-контейнер**, а не його елементи. Вкладений `Field` —
якщо це `record` зі скалярами — порівнюється коректно сам собою.

### 4.4. Рішення — тип-обгортка

Підхід, до якого дійшли:

- **`Field`** — звичайний `record` зі скалярами. Авторівність коректна, нічого
  робити не треба.
- **Колекція `fields`** — не «голий» масив, а **тип-обгортка** над
  `ImmutableArray<Field>`. Обгортка реалізує `IEquatable<T>` із **поелементним**
  `Equals` (семантика `SequenceEqual`) і **узгодженим** `GetHashCode`,
  порахованим за елементами.
- **Модель верхнього рівня** (`id` + обгортка) — знову звичайний `record`. Працює
  автоматично, бо всі її поля тепер коректно порівнюються.

Зауваження про `IEquatable`: `record`-и вже реалізують `IEquatable<T>` —
проблема не у відсутності інтерфейсу, а в тілі згенерованого `Equals`. `IEquatable`
доречний саме в **обгортці**, де типізований `Equals` пишеться руками один раз.

`ImmutableArray<T>` сам собою рівності за вмістом **не** дає — це поширена
помилка. Потрібен власний компаратор/`Equals` у обгортці.

### 4.5. Чому не серіалізація / Base64

Серіалізація моделі в рядок (зокрема Base64 від байтів) як ключ кешу —
*можлива*, але не «безкоштовна»: вимагає **детермінованої канонічної**
серіалізації. Це не менше роботи, ніж написати `Equals`, і складніше для
відлагодження: помилка в `Equals` видно одразу, а недетермінована серіалізація
дає *плаваючу* хибну інвалідацію. Base64 конкретно нічого не додає — проблема не
в «бінарне/текстове», а в посилальному vs значущому порівнянні.

### 4.6. Порядок елементів

Поелементне порівняння вважає `[a, b]` і `[b, a]` різними. Якщо порядок `fields`
семантично **не** важливий — варто сортувати за стабільним ключем (напр., за
`name`) під час парсингу, щоб переупорядкування в JSON не давало хибної
інвалідації. Якщо важливий — лишаємо як є, але свідомо. Це питання ортогональне:
не закривається ані `record`, ані обгорткою, ані серіалізацією.

### 4.7. `record` на `netstandard2.0`: `IsExternalInit`

`record` генерує init-only сетери, а для них компілятор потребує тип-маркер
`System.Runtime.CompilerServices.IsExternalInit`. У .NET 5+ він є в рантаймі, у
`netstandard2.0` — **немає**, тож компіляція падає з помилкою
«predefined type `IsExternalInit` is not defined».

Лікується підкладанням типу самотужки — порожній `internal static class
IsExternalInit` у namespace `System.Runtime.CompilerServices`, один файл на
проєкт. Namespace та ім'я мають бути точними — компілятор шукає за повним
ім'ям. Альтернатива — пакет `PolySharp`, але для одного типу окремий пакет —
надмір.

## 5. Генерований маркерний атрибут `[ServerLogic]`

### 5.1. Що це за атрибут і чому саме так названий

Атрибут позначає клас, який **перехоплює та розширює серверну логіку сутності**:
хуки життєвого циклу (`BeforeSave`, `AfterSave`), додавання команд
(`Commands.Add(...)`), і, ймовірно, ще щось — набір можливостей **відкритий**.

Назва `[ServerLogic]` обрана свідомо як **роль класу**, а не як технічна
механіка. Розглянуті й відкинуті варіанти:

- `[CompileMarker]` (стара назва) — називає технічну механіку («маркер для
  генератора»), а не роль; користувач не має думати про компілятор.
- `[ServerHook]` — називає лише одну зі здібностей (хуки) і посвариться з
  `Commands` та майбутніми можливостями.
- `[OnServer]` — `On...` несе подієву семантику («при настанні події»), а клас
  не реагує на подію, він *містить* логіку; до того ж називає «де», а не «що».

`[ServerLogic]` називає роль, а не здібність, тому переживе додавання нових
можливостей. Ім'я — частина публічного контракту: його бачить кожен, хто ставить
атрибут, і воно має пояснювати себе без документації.

### 5.2. Інжект атрибута — через `RegisterPostInitializationOutput`

Атрибут — **compile-time маркер** (у рантаймі не потрібен). Генератор **сам
додає** його визначення в компіляцію через `RegisterPostInitializationOutput`.

- Атрибут **не лежить** у жодній окремій збірці фізично — генератор інжектить
  його в кожен цільовий проєкт, до якого підключений.
- У цільовому проєкті це непомітно: підключили пакет із генератором — атрибут
  «з'явився», можна ставити `[ServerLogic]`. Жодного `ProjectReference` не треба,
  потрібне лише підключення проєкту генератора як аналізатора.
- `RegisterPostInitializationOutput` **не має доступу до провайдерів** —
  виконується до побудови pipeline. Туди йде лише статичний текст. Тому
  namespace атрибута **не може** залежати від проєкту-споживача (його просто
  немає звідки взяти на цьому етапі) — і не повинен (див. 5.4).

### 5.3. `global using` — щоб користувач писав коротко

Щоб у цільовому проєкті працювало просто `[ServerLogic]` без жодного `using`,
генератор інжектить **другим файлом** `global using` для свого namespace.

Чому **окремим** файлом: `global using` за правилами C# має стояти до оголошень
типів **у межах файла**, а визначення атрибута — це оголошення типу. Злити в
один файл не можна — директива опиниться після типу. Тому в
`RegisterPostInitializationOutput` два `AddSource`: один файл — атрибут, другий —
`global using`.

- Порядок двох `AddSource` між собою — **байдужий**: обидва файли потрапляють у
  компіляцію, `global using` діє на всю компіляцію незалежно від порядку.
- Порядок рядків у `Initialize` — функціонально байдужий (Roslyn гарантує, що
  post-init відпрацює до основного проходу), але post-init варто писати першим
  заради читабельності — метод читається згори вниз у напрямку потоку даних.
- Наслідок: `global using` протікає на **весь** проєкт-споживач — ім'я
  `ServerLogic` стає видимим глобально. Для унікального імені це прийнятно, але
  це свідоме втручання в глобальний простір імен чужого проєкту.

### 5.4. Рішення, які варто ухвалити свідомо

- **Namespace атрибута.** Визначає генератор; має бути **фіксованим**, не
  залежним від проєкту-споживача. Те саме повне ім'я використовується в трьох
  місцях: `namespace` у тексті атрибута, рядок-аргумент
  `ForAttributeWithMetadataName`, і `global using`. Усі три мають збігатися
  **буквально** — варто тримати namespace однією `const` і будувати решту з неї
  інтерполяцією, щоб джерело істини було одне фізично.
- **Видимість.** `internal` достатньо для використання лише в межах власного
  рішення; `public` — якщо генератор їде як NuGet назовні.
- **`[AttributeUsage(AttributeTargets.Class)]`** — обмежити застосування лише
  класами.
- **Партіальність.** Атрибут досить поставити на одному оголошенні класу — якщо
  `Agent` розбитий на кілька файлів, дублювати `[ServerLogic]` не треба.

### 5.5. Колізія атрибута між кількома збірками

Якщо генератор підключений до **кількох проєктів одного рішення, що посилаються
одне на одного**, `internal`-атрибут із 5.4 дасть конфлікт «тип визначено у двох
збірках». Штатне рішення цієї конкретної колізії — `AddEmbeddedAttributeDefinition`
(інжектить `Microsoft.CodeAnalysis.EmbeddedAttribute`) + позначення власного
атрибута `[Embedded]`: тоді тип стає невидимим за межами своєї збірки.
*(Потрібно, лише якщо сценарій — кілька пов'язаних проєктів. Для одного
проєкту-споживача або непов'язаних проєктів — не треба.)*

## 6. Згенерований код не покладається на налаштування проєкту

Наскрізне правило для **всього** інжектованого/згенерованого коду (атрибут,
`global using`, компаньйон, реєстр):

- **Повні імена з `global::`-префіксом.** Згенерований код потрапляє в чужий
  проєкт, де може бути власний тип `System`, власний `Attribute` тощо. `global::`
  гарантує резолвинг від кореня: `global::System.Attribute`, не голий `Attribute`.
- **Не покладатися на `ImplicitUsings`.** `ImplicitUsings` технічно *дістає* до
  згенерованого коду (SDK генерує `GlobalUsings.g.cs`, який діє на всю
  компіляцію), але склад implicit-usings залежить від SDK
  (`Microsoft.NET.Sdk` / `.Web` / `.Worker`) та його версії. Генератор не знає,
  у який проєкт його підключать, тож не повинен залежати від цього набору.
- **`#nullable enable` у самому файлі.** Щоб поведінка не залежала від
  nullable-налаштування проєкту-споживача.
- **`// <auto-generated/>` у шапці.** Глушить аналізатори та StyleCop на
  згенерованому файлі.

Спільна думка: згенерований код робить себе **незалежним** від налаштувань
проєкту-споживача, а не підлаштовується під них.

## 7. Генератор ассетів (генератор B)

### 7.1. Реєстрація виходу

Ембедер ассетів реєструє свій вихід через:

    context.RegisterImplementationSourceOutput(...)

замість звичайного `RegisterSourceOutput`. Це повідомляє Roslyn, що згенерований
код потрібен лише для фінальної збірки і може пропускатися під час design-time
(IDE) прогонів. Дороге стиснення таким чином виконується при реальній збірці й не
висить фоном у IDE.

Застосовно це лише тому, що на згенерований ассет-код користувацький код не
посилається типобезпечно (доступ — по ключу в рантаймі). Для строго типізованого
доступу `RegisterImplementationSourceOutput` не підійшов би.

### 7.2. Ідентифікатор ассета з відносного шляху

Ключ ассета в реєстрі — **відносний шлях від кореня проєкту**, де роздільники
замінено на `_`. Приклад: `catalog/agent/module.json` → `catalog_agent_module_json`
(розширення теж схлопується).

Обчислення розпадається на три кроки:

1. **Відняти корінь** від повного шляху ассета. Корінь — директорія проєкту;
   повний шлях ассета — з `AdditionalText.Path`. `Path.GetRelativePath` для цього
   **недоступний** на `netstandard2.0` (з'явився у .NET Standard 2.1) — або
   поліфіл (`PolySharp`), або власне віднімання префікса.
2. **Нормалізувати.** Обидва шляхи канонізувати **до** віднімання: єдиний
   роздільник, порівняння префікса через `OrdinalIgnoreCase` (регістр диска на
   Windows). Інакше віднімання мовчки промахнеться. Це та сама проблема
   нормалізації шляхів, що й у join розділу 3.4.
3. **Схлопнути в ідентифікатор.** Правило — **білий список**: лишаємо валідне для
   C#-ідентифікатора (`Char.IsLetterOrDigit` або `_`), решту → `_`. Білий список,
   а не чорний: пробіл, `+`, дужки тощо теж мають схлопнутися, а не протекти.

Свідомі рішення:

- **Стабільність між Windows і CI.** Якщо ключ має бути однаковим на машині
  розробника та на CI (Linux) — роздільники треба примусово нормалізувати, бо
  `\` і `/` дають різні рядки для того самого ассета.
- **Юнікод у іменах.** `Char.IsLetterOrDigit` пропускає кирилицю як «букву» —
  *вирішено лишити як є* (C# дозволяє юнікод в ідентифікаторах).
- **Колізія.** `catalog/agent/module.json` і `catalog/agent-module.json` дають
  той самий ключ. Білий список цього не вирішує. *(Залишилось відкритим — див.
  питання 3.)*
- **Провідний символ.** Якщо ключ іде в ім'я C#-члена і починається з цифри —
  невалідний; за потреби додавати префікс. Для шляхів зазвичай неактуально.

Корінь проєкту можна брати або через `CompilerVisibleProperty`/`ProjectDir`
(розділ 3.5), або обчислювати з якірного файла відомої глибини. Останнє легальне,
лише якщо глибина якоря **фіксована** — інакше «шукати корінь угору по дереву»
впирається в заборонену файлову систему (розділ 3.1).

## 8. `.csproj`: `AdditionalFiles`

- Файли збираються **точними glob'ами** за розширеннями (можна одним рядком через
  `;`, можна кількома — поведінка однакова). **Не** `**/*` — він тягне `.cs`,
  бінарники й увесь `obj` у пул без користі.
- `Include` приймає рядок або glob; `**` — рекурсивний; кілька `Include`
  накопичуються в одну колекцію.
- Для виключення `bin`/`obj` — `Exclude="$(DefaultItemExcludes)"`. Це штатне
  SDK-властивість, яке SDK обчислює пізно (вже після фіналізації
  `BaseOutputPath`/`BaseIntermediateOutputPath`), тому посилатися на нього
  безпечніше, ніж на ці шляхи напряму.
- `AdditionalFiles` — **спільний пул** на проєкт; розведення «потоків» живе не в
  `.csproj`, а у `.Where` всередині генераторів.

## 9. Відкриті питання (на майбутнє)

1. Правило зіставлення «клас ↔ його `metadata.json`»: строго одна папка = один
   клас = один JSON, чи можливі відсутні/множинні — і як тоді поводитися
   (діагностика? пропуск?). Впливає на тип на виході `Combine` (розділ 3.3).
2. Склад реєстру entity: усі позначені атрибутом типи, чи лише ті, для яких JSON
   знайшовся і компаньйон реально згенерувався.
3. Колізія ключів у реєстрі ассетів: `a/b/c.json` і `a/b-c.json` дають однаковий
   ключ — заборонити такі імена, чи розрізняти (напр., не схлопувати розширення)?
4. Порядок елементів у `fields`: сортувати при парсингу чи лишати як є.

## Додаток. Закриті в цій сесії питання

- **Namespace атрибута з шляху проєкту** — питання знято: задача була
  переосмислена. Namespace атрибута має бути фіксованим (5.4), обчислювати його
  з `ProjectDir` і не потрібно, і неможливо (post-init не бачить провайдерів).
  Namespace компаньйона береться з самого позначеного класу (`ContainingNamespace`),
  namespace реєстру — за потреби з `RootNamespace` через `CompilerVisibleProperty`.
- **Перейменування `[CompileMarker]` → `[ServerLogic]`** — зі зміною трактування:
  не «технічний маркер для генератора», а **роль класу** (серверна логіка
  сутності). Генерація компаньйона — наслідок ролі, не сама роль.
